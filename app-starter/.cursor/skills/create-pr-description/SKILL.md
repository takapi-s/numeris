# PR Description 生成

## 概要

現在のブランチとベースブランチ（main/master）との差分を分析し、プロジェクトのPRテンプレートに準拠したdescriptionを生成する。

## ワークフロー

### Step 1: 差分情報の収集

以下のコマンドを実行して差分を把握する：

```bash
# ベースブランチを特定（main または master）
git rev-parse --verify main 2>/dev/null && BASE=main || BASE=master

# ブランチの差分サマリー
git diff $BASE...HEAD --stat

# コミット履歴
git log $BASE..HEAD --oneline

# 詳細な差分（必要に応じて）
git diff $BASE...HEAD
```

### Step 2: 変更の分析

収集した情報から以下を特定する：

- **変更の目的**: 新機能/バグ修正/リファクタリング/ドキュメント更新など
- **影響範囲**: 変更されたファイル・モジュール
- **破壊的変更の有無**: API変更、データベーススキーマ変更など
- **UI/UX変更の有無**: フロントエンドの見た目に関わる変更

### Step 3: Description の生成

以下のテンプレートに従って記述する：

```markdown
# Pull Request

## 概要
<!-- 1〜2文で変更の目的と背景を説明 -->
[変更の目的を簡潔に記述]

## 変更内容
<!-- コミットやdiffから主要な変更を箇条書きで記載 -->
- [変更点1]
- [変更点2]
- [変更点3]

## 関連するIssueやドキュメントがあれば記載してください
<!-- Issue番号が特定できる場合のみ記載 -->
Closes #[issue番号]

## スクリーンショット・動画
<!-- UI/UX変更がある場合のみ記載を促す -->
<!-- 変更がない場合は「該当なし」と記載 -->

## チェックリスト
- [ ] コードレビューの準備ができている
- [ ] ドキュメントを更新した（必要に応じて）
- [ ] 破壊的変更がある場合、マイグレーション手順を記載した
- [ ] UT、もしくは手動テストを実施した

## 追加情報
<!-- 実装の背景、技術的な判断理由、参考にしたリソースなど -->
[必要に応じて記載]

## レビュアーへのメモ
<!-- 重点的に見てほしい箇所、議論したい設計判断など -->
[必要に応じて記載]
```

## 記述ガイドライン

### 概要セクション

- **目的**: 「なぜこの変更が必要か」を1〜2文で
- **避けるべき**: コミットメッセージの羅列、技術詳細の列挙

**良い例**:
> ユーザー認証フローにおいてセッションタイムアウト後の再ログイン体験を改善し、作業中のデータが失われないようにした。

**悪い例**:
> AuthService.tsを修正、SessionManager.tsを追加、login.tsxを更新

### 変更内容セクション

- 技術的な変更を**機能的な観点**で記述
- 重要度の高い順に並べる
- 必要に応じてサブリストで詳細を補足

**良い例**:
```markdown
- セッション延長機能を追加
  - バックグラウンドでのトークンリフレッシュ
  - 有効期限5分前の自動延長
- ログイン画面にリダイレクト先の保持機能を追加
- セッション関連のエラーハンドリングを統一
```

### レビュアーへのメモ

レビュー効率を上げるための情報を記載：

- 特に注意して見てほしいファイル・関数
- 設計上の判断で議論したい点
- 既知の制限事項や今後の改善予定

## 注意事項

- Issue番号が不明な場合は `Closes #` の行を削除するか空欄にする
- UI変更がない場合はスクリーンショットセクションに「該当なし」と記載
- 破壊的変更がある場合は**必ず**追加情報セクションでマイグレーション手順を記載
